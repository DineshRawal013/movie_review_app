# DEF-001 — Cookie SameSite attribute is "lax" not "strict" (Auth tokens)

**Defect ID:** DEF-001
**Severity:** P1 — Critical
**Status:** FIXED
**Raised by:** qa-engineer
**Date Raised:** 2026-05-26
**Phase Detected:** 7 — Test (Static analysis)
**Assigned to:** backend-engineer

---

## Requirement Violated

- NFR-020 — Auth tokens in httpOnly cookies; cookie flags must include SameSite=Strict.
- TC-AUTH-003 (step 4): Verify `access_token` and `refresh_token` cookies have `SameSite=Strict`.

---

## Description

In `backend/src/auth/auth.service.ts`, the `issueTokenPair()` method sets cookie options with:

```typescript
sameSite: 'lax' as const,
```

Both the `access_token` and `refresh_token` cookies (and the `csrf_token` cookie) are issued with `SameSite=Lax` rather than `SameSite=Strict`.

The TC-AUTH-003 test case (step 4) and NFR-020 acceptance criterion explicitly require `SameSite=Strict` for all auth cookies. `SameSite=Lax` permits cookies to be sent on top-level cross-site GET navigations (e.g., following a link from another site), which expands the attack surface compared to `Strict`.

---

## Steps to Reproduce

1. Authenticate via Google OAuth callback.
2. Inspect `Set-Cookie` response headers for `access_token` and `refresh_token`.
3. Observe: `SameSite=Lax` instead of `SameSite=Strict`.

---

## Expected Result

```
Set-Cookie: access_token=...; HttpOnly; Secure; SameSite=Strict; Path=/; Max-Age=900
Set-Cookie: refresh_token=...; HttpOnly; Secure; SameSite=Strict; Path=/; Max-Age=604800
```

---

## Actual Result

```
Set-Cookie: access_token=...; HttpOnly; Secure; SameSite=Lax; Path=/; Max-Age=900
Set-Cookie: refresh_token=...; HttpOnly; Secure; SameSite=Lax; Path=/; Max-Age=604800
```

---

## Fix Guidance

In `backend/src/auth/auth.service.ts`, change the `cookieOpts` object:

```typescript
// Before
sameSite: 'lax' as const,

// After
sameSite: 'strict' as const,
```

Also update the `csrf_token` cookie's `sameSite` to `'strict'` on line 127.

---

## Impact Assessment

- Security exposure: Low-to-moderate additional risk vs Strict; not a complete bypass.
- P1 because NFR-020 explicitly calls for Strict and this fails the TC-AUTH-003 acceptance criterion.
- Does NOT block P0 security requirement (tokens remain httpOnly, Secure).
- Fix is a one-line change with zero risk of regression.

---

## Regression Risk

Low. Changing `lax` to `strict` only restricts when the cookie is sent; it does not break any legitimate application flow (the app uses same-site navigation throughout).
