# DEF-002 — GDPR account deletion performs hard-delete instead of soft-delete on user row

**Defect ID:** DEF-002
**Severity:** P2 — Major
**Status:** OPEN
**Raised by:** qa-engineer
**Date Raised:** 2026-05-26
**Phase Detected:** 7 — Test (Static analysis)
**Assigned to:** backend-engineer

---

## Requirement Violated

- TC-AUTH-011 (step 5): "Query `users`: verify `deleted_at` is set for Alice's row (soft delete)."
- TC-AUTH-011 (step 11): "Attempt GET `/api/auth/google/callback` with Alice's Google ID ... verify new account creation is allowed (deleted_at does NOT prevent re-registration)."
- LLD Section 3.5 (UsersService.deleteOwnAccount): specifies user row is soft-deleted (`deleted_at` set); associated data hard-deleted.

---

## Description

In `backend/src/users/users.service.ts`, the `deleteAccount()` method uses `tx.user.delete()` (hard delete) rather than `tx.user.update({ data: { deletedAt: new Date() } })` (soft delete):

```typescript
async deleteAccount(id: string) {
  return this.prisma.$transaction(async (tx) => {
    await tx.refreshToken.deleteMany({ where: { userId: id } });
    await tx.reviewFlag.deleteMany({ where: { userId: id } });
    await tx.rating.deleteMany({ where: { userId: id } });
    await tx.review.deleteMany({ where: { userId: id } });
    await tx.user.delete({ where: { id } });   // <-- hard delete
  });
}
```

The test case (TC-AUTH-011) and LLD explicitly require the user row to be soft-deleted (`deleted_at` set), while associated data (reviews, ratings, tokens, flags) should be hard-deleted. A hard delete of the user row prevents audit trail reconstruction and may cause FK constraint issues if `audit_log` rows reference the user's `id`.

---

## Steps to Reproduce

1. Create user Alice; authenticate.
2. Call DELETE `/api/users/me` with Alice's auth.
3. Query `users` table: expect `deleted_at` set on Alice's row.
4. Actual: Alice's row is completely absent from the table.

---

## Expected Result

- `users` row: `deleted_at = NOW()` (soft delete preserved for audit trail).
- All associated rows (reviews, ratings, refresh_tokens, review_flags): hard-deleted.

---

## Actual Result

- `users` row: completely deleted (hard delete).
- Associated rows: hard-deleted (correct).

---

## Fix Guidance

In `backend/src/users/users.service.ts`, change the last line of the transaction:

```typescript
// Before
await tx.user.delete({ where: { id } });

// After
await tx.user.update({ where: { id }, data: { deletedAt: new Date() } });
```

Ensure the `users` Prisma model has a `deletedAt DateTime?` field mapped to the `deleted_at` column in the schema.

---

## Impact Assessment

- Not a P1 because the user-visible behaviour (account deletion, data erasure) still functions correctly from the user's perspective.
- Audit trail gap: admin cannot see that the Google ID `google-123` previously existed.
- Re-registration: with a hard delete, re-registration of the same Google ID works via upsert — but if the Google ID row is gone, `findUnique` returns null and `create` is used. This path actually works, but the LLD intent was to track deletions.
- FK concern: `audit_log.admin_user_id` and `audit_log.target_user_id` may reference the deleted user UUID, causing dangling references (depending on DB CASCADE config).

---

## Regression Risk

Low. Changing to soft-delete requires the Prisma schema to have `deletedAt` on the `User` model and a corresponding migration to add the column if not already present. Verify ERD.md to confirm `deleted_at` column exists (it is defined in ERD.md Section 2.1).
