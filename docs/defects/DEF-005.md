# DEF-005 — CreateReviewDto uses "content" field but api-spec.yaml specifies "body"

**Defect ID:** DEF-005
**Severity:** P2 — Major
**Status:** FIXED
**Raised by:** qa-engineer
**Date Raised:** 2026-05-26
**Phase Detected:** 7 — Test (Static analysis / API contract review)
**Assigned to:** backend-engineer

---

## Requirement Violated

- api-spec.yaml: POST `/movies/{tmdbId}/reviews` request body schema defines field `"body"` (string, max 500 chars).
- TC-REVIEWS-005 (step 1): `POST /api/movies/{movieId}/reviews` with `{ "body": "This is a great film." }`.
- FR-050 — Review body field is the text content of the review.

---

## Description

`backend/src/reviews/dto/create-review.dto.ts` defines the DTO field as `content`:

```typescript
export class CreateReviewDto {
  @IsString()
  @IsNotEmpty({ message: 'Review text cannot be blank' })
  @MaxLength(500, { message: 'Review cannot exceed 500 characters' })
  content: string;   // <-- named "content", not "body"
}
```

However, the approved api-spec.yaml specifies the JSON request body field as `"body"`. A client sending `{ "body": "My review" }` per the spec will receive a 422 Unprocessable Entity because the DTO expects `content`, not `body`.

Additionally, `reviews.service.ts` creates the DB record as `body: dto.content`, confirming the internal mapping but showing the external interface discrepancy.

The same issue applies to `update-review.dto.ts` (assumed to also use `content`).

---

## Steps to Reproduce

1. Send `POST /api/movies/550/reviews` with body `{ "body": "Great film!" }` (per api-spec) and valid auth.
2. Observe: HTTP 422 — validation fails because `content` is required and not provided.

---

## Expected Result

- `POST /api/movies/{id}/reviews` with `{ "body": "Great film!" }` returns HTTP 201.

---

## Actual Result

- HTTP 422: `content should not be empty`.

---

## Fix Guidance

Option A — Rename DTO field to match the spec (recommended):
```typescript
// create-review.dto.ts
export class CreateReviewDto {
  @IsString()
  @IsNotEmpty({ message: 'Review text cannot be blank' })
  @MaxLength(500, { message: 'Review cannot exceed 500 characters' })
  body: string;
}
```
Update `reviews.service.ts` line 60: `body: dto.body`.
Update `update-review.dto.ts` and `reviews.service.ts` line 71 correspondingly.

Option B — Add `@ApiProperty` with `@Transform` to accept both; not recommended (adds complexity).

---

## Impact Assessment

- Any API client following the api-spec.yaml (including integration tests written to spec) will fail to submit reviews.
- Frontend `write-review.tsx` calls `reviewsApi.create(movieId, { content })` — uses `content` not `body` — meaning the frontend was built against the DTO, not the spec. If Option A is applied, the frontend `lib/api.ts` also needs to be updated to send `{ body: content }`.
- P2 not P1: the functionality works end-to-end if the client uses the DTO field name, but the published API contract is violated.

---

## Regression Risk

Medium. Renaming the field requires updating DTO, service, and frontend API call. All changes are localized and low-risk individually.
